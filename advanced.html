A2 Cache placement 15 min

cache_control is a watermark: everything before it (in token order) is cached. If you put it on a tool definition, the system prompt and all preceding tools are cached too.

Task: Run two experiments from Section A2 of caching.py: (1) cache only the system prompt, (2) cache a tool definition. Compare which placement caches more tokens on a typical agent loop call.

Observe this

In an agent loop where the system prompt is 200 tokens and tool definitions are 300 tokens, which placement gives better cache utilization per API call? What's the right placement strategy for an agent that runs 20+ steps?

A3 Agent loop caching 20 min

In a multi-step loop, the system prompt and tool definitions are byte-for-byte identical on every iteration. Without caching, you pay for them on every call. With caching, after step 1 you only pay for the new messages.

Task: In Section A3 of caching.py, run run_agent_loop() with use_cache=False and then with use_cache=True. Compare total input tokens and total cache-read tokens across all steps.

Observe this

After a 5-step loop, what % of total input tokens were served from cache? Now extrapolate to a 20-step loop. At what point does the cache payoff become significant?

B1 Reflection loop 20 min

Draft → critique → revise, repeated until the model rates its own output as good or a max iteration count is hit. The stopping criterion prevents infinite loops.

Task: Complete reflection_loop() in patterns.py. The critique phase must ask for a good/needs-work rating. If good, stop. If needs-work, send the critique to a revise call and update the draft.

Observe this

How often does the model say "good" on the first try vs after revisions? Run the same task 3 times. Is the number of revision cycles consistent? What does this tell you about when reflection is worth the extra token cost?

B2 Parallel fan-out 25 min

Fork work to N independent subagents running in parallel. Each has its own context. Merge their results with a final synthesis call. Use threading.Thread — it's simpler than asyncio for this pattern.

Task: Complete the parallel fan-out in Section B2 of patterns.py. Three threads run simultaneously, one per framework (Django, FastAPI, Flask). After all threads complete, merge results with one synthesis call.

B3 Router pattern 20 min

A fast, cheap model classifies the user request and routes it to the right specialist. The router itself does no domain work — classification only.

Task: Complete the specialist agents in Section B3 of patterns.py (research_agent, code_agent, summarize_agent). Then run route() on 8 diverse inputs and log which specialist was called.

Observe this

Is there a category of input that gets misrouted consistently? What changes to the classifier system prompt fix it — and do those changes cause different misroutes elsewhere? This is the core challenge of router design.

B4 Human-in-the-loop 15 min

The agent has an ask_human(question) tool. When it calls it, your code calls input() and returns the human's typed response as a tool_result. The agent then continues with that answer in context.

Task: Complete the HITL agent loop in Section B4 of patterns.py. Test it with the task: "Write a summary of machine learning to ml_summary.txt, but ask the human how detailed it should be before writing."

Observe this

Does the agent ask the question before writing — or does it write anyway and treat the tool call as optional? What system prompt phrasing reliably causes it to pause vs proceed? This gap between description and behavior is a fundamental challenge in agentic systems.

Section C · ~1.5 hours

Subagents vs Skills

Learn when a function is enough, when a class is better, and when you actually need a Claude call.

Quick win · 10 min

Cost comparison in real numbers

Run the warm-up in skills.py. It solves the same task two ways: a direct Python function (0 tokens, <1ms) and a Claude subagent call (~50 tokens, ~1s). The numbers tell you the tradeoff immediately.

Subagents cost tokens and latency on every single call. That's not a reason to never use them — it's a reason to only use them when you actually need language understanding.

C1 The decision framework 15 min

Four levels: tool (pure Python), skill module (reusable class wrapping tools), subagent (separate Claude call), independent agent (full autonomous loop).

Task: Complete Section C1 in skills.py — categorize each capability in the list. For each, write a one-sentence justification. There are no trick answers — some genuinely sit on the boundary.

Observe this

Which capabilities are genuinely ambiguous — where both tool and subagent could work? Write down the specific condition that tips you toward one or the other. That condition is your personal decision rule.

C2 Skill modules 20 min

A skill is a Python class that bundles related tools with their schemas and error handling. SearchSkill and FileSkill are already written in skills.py. Your job: implement MemorySkill.

Task: Implement MemorySkill with a tool_definition property and remember(fact) / recall_all() methods. The skill stores facts in self.facts (a list). Test all three skills at the bottom of Section C2.

Observe this

What changes in your agent loop code when you switch from inline tool functions to skill classes? What's easier to test now — and what gets slightly more verbose? Is the tradeoff worth it at the scale of a session's codebase?

C3 Subagent specialization 20 min

ResearcherAgent (search + memory) and WriterAgent (file I/O only) are partially implemented. Giving each agent only the tools it needs keeps their context small and their behavior predictable.

Task: Implement MemorySkill first (C2), then wire it into ResearcherAgent. Run a two-phase pipeline: researcher gathers facts, writer turns them into a document. Log context sizes at each step.

Observe this

Compare context size: how many messages does the researcher accumulate vs the writer? If you'd used a single agent for both phases, how much larger would that context be? Does the split improve or hurt the final output quality?

D1 Tools via MCP 20 min

Tools in MCP are just decorated Python functions. @mcp.tool() exposes them in the protocol's JSON-RPC interface. Clients discover them automatically via a tools/list call.

Task: Add a word_count(text: str) -> str tool to mcp.py using @mcp.tool(). Run python mcp.py --test and confirm it works. Then connect the server to Claude Code as a local MCP server and call it from there.

Observe this

In Claude Code's tool list, how does your MCP tool's description and schema appear compared to how you defined it in Python? Is there anything that got lost or changed in translation — or is it exactly what you wrote?

D2 Resources 20 min

Resources are read-only endpoints — think of them as URLs the model can fetch. Your server already exposes the Acme KB chunks as resources. Add one more.

Task: Add a logs://agent/latest resource to mcp.py that reads agent.log (from Session 3) if it exists, or returns "No logs found.". Test it with python mcp.py --test.

Observe this

When would you use a resource instead of a tool? The key difference: tools are actions (model calls them when it needs to do something), resources are data (model fetches context). Think of a case from Sessions 1–6 that would have been better modeled as a resource.

D3 Prompts 15 min

Prompts in MCP are reusable templates with parameters. Any client can list them and invoke them — you get consistent prompt engineering shared across tools and agents.

Task: Add a compare_options(a: str, b: str) prompt to mcp.py using @mcp.prompt(). It should generate a structured comparison prompt for any two options. Test it in Claude Code by calling it through the MCP interface.

Observe this

If you have 3 agents that all need to do structured comparisons, what changes with MCP prompts vs inline strings? Think about what happens when you want to improve the prompt — how many files change in each approach?

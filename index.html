Session 7 · ~3–4 hours

Capstone

Design and ship something real.

Quick win · 10 min

Design before you code

Before opening your editor, open a text file (or grab paper). Write answers to these five questions — actual answers, not "TBD":

1. Which architecture pattern from Session 4?
2. What external tools does it need?
3. Where does memory live — in-context, rolling summary, or vector store?
4. What are the three most likely failure modes?
5. What gets logged, and what would you look at first in a failed run?

If you can't answer all five in 10 minutes, the design isn't ready. That's not a failure — that's the warm-up doing its job.

7A Written design 20 min

Save your design to DESIGN.md before you write any code. It doesn't need to be long — it needs to be decided.

Cover: which pattern and why you chose it over the others; which tools are needed and whether any are stateful; where memory lives and at what token threshold compression fires; the max_steps and token budget limits; what events get logged.

Observe this

What was the hardest design decision? Write down the tradeoff explicitly — what you chose and what you gave up. That tradeoff is a learning outcome, not just an implementation detail.

7B Build it 60 min

Implement your design. Required checklist:

☐ At least one real external tool (not faked)
☐ Context compression at a defined threshold
☐ max_steps and token budget enforced
☐ Structured JSON log file
☐ Retry with exponential backoff on API calls

Observe this

You will hit at least one decision mid-build that wasn't in your design. Write it down when it happens. Those surprises — the things the design didn't anticipate — are the real output of this chunk.

7C Document it 25 min

Write two documents: a one-page ARCHITECTURE.md and a README.md with setup instructions.

The architecture doc covers: which pattern and why, where memory lives, how compression works, the failure modes you designed for, and one thing you'd do differently if rebuilding from scratch.

Observe this

Answer this in 30 seconds: if someone asked you to add a fact-checking sub-agent to your system — where would it plug in? If you can answer immediately without re-reading your code, you understand your own architecture.
